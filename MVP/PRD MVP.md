# nuevo MVP PRD

**문서 목적**: 지금까지 만든 프로토타입과 분리해서, 실제로 작동하는 MVP를 개발하기 위한 제품 요구사항만 정의한다.  
**MVP 슬로건**: **올리고 → 바로 놀아보는**  
**MVP 제외**: 플랫폼 안에서 AI로 직접 “만들기” 기능은 제외한다. 창작자는 이미 외부 도구로 만든 앱 URL을 올린다.

---

## 1. MVP 한 줄 정의

AI로 만든 웹 앱 URL을 창작자가 업로드하면, 다른 사용자가 모바일 피드 안에서 페이지 이동 없이 바로 실행하고 반응할 수 있는 서비스.

---

## 2. MVP 핵심 가설

Instagram Reels에는 AI로 만든 재미있는 앱을 영상으로 찍어 올리는 콘텐츠가 많다.

nuevo MVP는 이 영상 공유를 앱 직접 체험으로 바꾼다.

> “이거 영상으로 보는 대신, 여기서 바로 해볼 수 있으면 더 좋다.”

이 가설을 검증하기 위해 MVP는 창작/생성 기능보다 **URL 업로드, iframe 실행, 피드 체험, 반응 저장**에 집중한다.

---

## 3. 대상 사용자

### Creator

- Claude Code, Cursor, Replit, v0, Bolt 등으로 작은 앱을 만든 사람
- 만든 앱을 Instagram/TikTok/X에 영상으로 공유하는 사람
- 자신이 만든 앱을 사람들이 직접 써보길 원하는 사람

### Viewer

- AI 앱/실험/밈을 직접 만져보는 것을 좋아하는 사람
- 피드에서 가볍게 발견하고 바로 실행해보고 싶은 사람

---

## 4. MVP 핵심 루프

```text
Creator
회원가입 → 앱 URL 업로드 → 피드 노출

Viewer
피드 발견 → 카드 터치 → 인라인 확장 → iframe 실행 → 반응/댓글/팔로우

Creator
체험/댓글/반응 확인 → 다음 앱 업로드
```

---

## 5. MVP 기능 범위

### 5.1 인증

| 기능 | 요구사항 |
|---|---|
| Google 로그인 | Supabase Auth 기반 OAuth |
| 프로필 설정 | handle, display name, avatar 저장 |
| 보호 라우트 | 업로드, 스튜디오, 설정, 내 프로필은 로그인 필요 |

### 5.2 업로드

| 기능 | 요구사항 |
|---|---|
| 콘텐츠 타입 | MVP 핵심은 `interactive`; `audio`, `image`는 프로토타입 UI 유지 가능 |
| 앱 URL 입력 | `https://` 공개 URL 권장, `localhost`는 로컬 테스트 허용 |
| URL 검증 | `http`, `https` 외 protocol 차단 |
| iframe 미리보기 | 게시 전 업로드 화면에서 실행 가능 여부 확인 |
| 실패 안내 | iframe이 오래 로딩되면 새 탭 열기 액션 제공 |
| DB 저장 | `posts` 테이블에 저장 |

### 5.3 피드

| 기능 | 요구사항 |
|---|---|
| 실 데이터 피드 | Supabase `posts` 조회 |
| mock fallback | DB가 비어 있거나 실패하면 프로토타입 mock 표시 가능 |
| 인라인 확장 | 카드 터치 시 상세 페이지 이동 없이 해당 카드 영역 확장 |
| iframe 실행 | 확장 영역 상단에서 앱 즉시 실행 |
| 실패 복구 | iframe 실패/지연 시 새 탭 열기 제공 |

### 5.4 소셜 액션

| 기능 | 요구사항 |
|---|---|
| 댓글 | 실 DB 포스트 기준 `comments` 저장/조회 |
| 반응 | `funny`, `weird`, `genius`, `wtf` 중 1개 저장 |
| 팔로우 | 로그인 유저의 follow 관계 저장 |
| 저장 | MVP에서는 로컬 저장 유지 가능 |

### 5.5 체험 지표

| 기능 | 요구사항 |
|---|---|
| 세션 시작 | iframe load 시 `experience_events` 생성 |
| 체험 시간 | unmount/visibility hidden 시 duration 업데이트 |
| 기본 스튜디오 | 내 작품별 체험 세션/시간 표시 |

### 5.6 리믹스

| 기능 | 요구사항 |
|---|---|
| 리믹스 버튼 | `/upload?remix={postId}` 이동 |
| 원본 연결 | `posts.remix_of` 저장 |
| 리믹스 알림 | 원본 창작자에게 알림 생성 |

## 6. MVP 성공 기준

MVP는 아래 시나리오가 실제 배포 URL에서 성공하면 “작동한다”고 본다.

1. 새 사용자가 Google로 로그인한다.
2. 프로필을 설정한다.
3. 외부에 배포된 앱 URL을 업로드한다.
4. 홈 피드에 새 포스트가 노출된다.
5. 다른 사용자가 피드에서 카드를 터치한다.
6. 앱이 인라인 iframe으로 실행된다.
7. 다른 사용자가 댓글/반응/팔로우를 남긴다.
8. DB에 댓글/반응/팔로우/체험 이벤트가 저장된다.
9. 창작자가 스튜디오에서 체험 수치를 확인한다.

---

## 7. Non-goals

MVP에서 하지 않는다.

- 앱 안에서 AI로 직접 앱 만들기
- 코드 생성/자동 배포
- 실제 광고 송출
- 실제 송금/정산
- 복잡한 추천 알고리즘
- 풀스케일 관리자 페이지
- 네이티브 앱
- 데스크톱 최적화

---

## 8. 운영 준비 요구사항

- Supabase schema가 실제 프로젝트에 적용되어야 한다.
- Google OAuth redirect URL이 로컬/배포 도메인 모두 등록되어야 한다.
- Vercel 환경변수가 설정되어야 한다.
- 최소 3개 이상의 실제 iframe 가능한 앱 URL을 seed content로 등록한다.
- `npm run lint`, `tsc`, `npm run build`가 통과해야 한다.

---

## 9. 프로토타입과의 관계

기존 프로토타입은 UI/UX와 visual concept 검증용이다. MVP는 프로토타입의 디자인 감각을 유지하되, 아래를 우선한다.

- mock보다 실 DB
- 넓은 기능보다 업로드/실행 루프
- 예쁜 데모보다 실제 사용 가능한 iframe
- 수익 UI보다 체험 이벤트 저장
